The resolution model has three core objects: `CompilePlan` (target + dependency plan), `PreparedProjectWorkspace` (materialized source roots and lockfile state), and resolved source input (`source_path`, `source`) used by compile commands.

`beskid_analysis::projects` owns graph building and manifest semantics; CLI commands and LSP integrations consume this model and should not reimplement manifest traversal rules.

**`Meta` projects** appear as ordinary graph nodes with **`type = Meta`** (see **[Project manifest contract](/platform-spec/tooling/manifests-and-lockfiles/project-manifest-contract/)**). Resolution must retain enough metadata for the **meta host** to register **event piping** into declared **`meta`** entry modules without conflating `Meta` roots with executable entrypoints unless the workspace explicitly composes them.
