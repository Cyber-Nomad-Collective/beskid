<SpecSection title="What this feature specifies" id="what-this-feature-specifies">
`Project manifest contract` defines one operational contract that a newcomer can follow end-to-end: first the model, then execution flow, then strict guarantees, concrete examples, and verification guidance.
</SpecSection>

<SpecSection title="Implementation anchors" id="implementation-anchors">

- Workspace resolution in `compiler/crates/beskid_analysis/src/resolve/items.rs`
- Contract tests in `compiler/crates/beskid_tests/src/projects/corelib/mod.rs`
</SpecSection>

<SpecSection title="Meta project type (`Meta`)" id="meta-project-type">
A manifest may declare **`type: Meta`** (exact field path and spelling are normative in schema articles below). A **`Meta`** project **does not** replace ordinary executable or library targets; it declares that this project participates in **compile-time meta orchestration**: the toolchain loads it so **`meta`** items can subscribe to **compilation events** and run **capture → process → emit** pipelines against host compilations, per **[meta compilation block](/platform-spec/language-meta/metaprogramming/meta-block/)** and **[Metaprogramming Mod SDK](/platform-spec/compiler/metaprogramming-mod-sdk/)**.

Schema fields for **which compilations** a `Meta` project attaches to, **entry modules**, and **budget defaults** are specified in the articles under this feature hub (see **contracts** and **examples**).
</SpecSection>
