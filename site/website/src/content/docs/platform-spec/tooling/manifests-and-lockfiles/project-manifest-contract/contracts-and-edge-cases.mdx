## What this article covers
This article explains the `contracts-and-edge-cases` view for `Project manifest contract` in concrete, onboarding-first language.

## Meta project contracts
- **`Meta` must be explicit** — If any compilation unit in a project contains a **`meta`** item and that project participates in workspace orchestration, its `project.type` **must** be `Meta`. Otherwise the host **must** emit a deterministic error whose code is registered alongside other manifest diagnostics.
- **Invalid `Meta` manifests are stable diagnostics** — Same inputs, same paths, same resolver version, and same host options yield the same diagnostic codes and primary messages (align with `compiler/crates/beskid_analysis` diagnostic tables and LSP parity rules).
- **Circular attachment** — `attachTo` graphs **must** be acyclic when interpreted as edges from `Meta` projects to host members. Cycles are deterministic resolution errors.
- **`entryModules` existence** — Each listed module path **must** resolve under the project `root`; missing files are errors at manifest validation time (not deferred to first compile).
- **Non-`Meta` projects** — Unless a future workspace opt-in is standardized, **`meta` in App/Lib/Test projects is forbidden** (see design model). Tooling **must not** silently ignore `meta` in forbidden contexts.

## Lockfile and registry interaction

`Meta` projects declare **`dependency`** blocks like any project. Compile-time execution **must** honor the same **locked** versions as the host compilation they attach to when both share a workspace lockfile scope. Publishing or CI that snapshots manifests **must** include `Meta` projects so mod graphs are reproducible.
