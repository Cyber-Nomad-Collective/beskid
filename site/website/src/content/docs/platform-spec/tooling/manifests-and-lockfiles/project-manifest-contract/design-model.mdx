## What this article covers
This article explains the `design-model` view for `Project manifest contract` in concrete, onboarding-first language.

## Meta project type (`Meta`)

`Project.proj` carries a **project kind** / **type** field on the `project` block. When set to **`Meta`**, the manifest denotes a project whose primary role is **compiler mod orchestration**: the workspace loads this project so its **`meta`** items (same surface syntax as language-meta **`meta`**) attach to **host** App/Lib/Test compilations and run **capture → process → emit** under host policy. A `Meta` project **does not** supply a normal JIT/AOT executable entrypoint unless the workspace also declares a conventional target graph that composes with tooling policy.

User-visible semantics of **`meta`** blocks remain in **[meta compilation block](/platform-spec/language-meta/metaprogramming/meta-block/)**; this contract defines **manifest shape** and **validation** so resolution, lockfiles, and the Mod SDK host agree on inputs.

### On-disk nesting under `project`

In `Project.proj`, **`meta`** configuration lives in a **nested block** under the top-level `project` block (YAML indentation style), for example:

```text
project:
  name: ...
  type: Meta
  meta:
    attachTo: ...
    entryModules: ...
```

Hosts **must** parse this as part of the `project` aggregate, not as a free-floating top-level block kind, so discovery stays consistent with `compiler/crates/beskid_analysis/src/projects/parser.rs` evolution. Unknown keys inside `meta` **must** error unless listed in the reserved-keys table in **[Workspace and lock contracts / design model](/platform-spec/tooling/manifests-and-lockfiles/workspace-and-lock-contracts/design-model/)**.

### Validation phases

1. **Structural parse** — While building `ProjectManifest`, reject duplicate keys, illegal types, and unknown `meta` keys.
2. **Graph validation** — When inserting the project into the workspace DAG, resolve `attachTo` entries against `Workspace.proj` members; detect cycles and unknown ids here so diagnostics include workspace context.

Each phase **must** use distinct stable diagnostic codes from the **E1801–E1899** meta/manifest band documented in **[Diagnostic code registry](/platform-spec/compiler/semantic-pipeline/diagnostic-code-registry/)**.

### Normative `project.meta` shape (v0.2 target)

Hosts **must** validate the following keys when `type: Meta` (field name `type` on `project`; value `Meta`). Unknown keys are rejected with stable diagnostics unless explicitly reserved for forward compatibility.

| Key | Required | Meaning |
| --- | --- | --- |
| `attachTo` | yes | Non-empty list of attachment selectors. Each entry is either the reserved word **`default`** (resolved only as documented in **[Workspace resolution / design model](/platform-spec/compiler/resolution-and-projects/workspace-resolution-contract/design-model/)**) or a **string member id** that must match a `workspace.members[].name` the resolver can prove exists. |
| `entryModules` | yes | Non-empty list of Beskid module paths (POSIX-style paths under the project `root`) that contain discoverable top-level **`meta`** items. At least one module **must** parse successfully for meta registration to succeed. |
| `maxMetaRounds` | no | Positive integer cap on host meta rounds per host compilation generation; default **`4`** when omitted. Exhaustion **must** surface a **compiler-native** error diagnostic, not a silent partial program. |
| `capabilities` | no | Closed vocabulary list granting compile-time powers (filesystem read roots, registry fetch, FFI bridges). Omitted means **minimal** capability set defined in **[meta block host bridge](/platform-spec/compiler/metaprogramming-mod-sdk/meta-block-host-bridge/)**. |

Legacy single-field `entryModule` (singular) **may** be accepted as an alias for a one-element `entryModules` list in transitional tooling; new manifests **should** use `entryModules`.

### Non-`Meta` projects and `meta` items

**Default platform policy (v0.2):** `meta` items **must not** appear in App/Lib/Test projects. Violation is a **deterministic diagnostic** (same code on CLI and LSP). There is **no** `Workspace.proj` opt-in field in v0.2; relaxing this rule **requires** a new normative workspace key and a minor spec bump—until then hosts **must** reject or reserved-error any unknown top-level workspace key that attempts to allow host `meta`.

`Meta` projects are the **supported** carrier for compiler mods. Documented exceptions (for example compiler-repo self-tests) **must** name the relaxation beside the example manifest that uses it.

## Practical guidance
When changing behavior in this area, update both the contract text and the corresponding compiler/runtime or tooling tests in the same change so the specification remains executable.
