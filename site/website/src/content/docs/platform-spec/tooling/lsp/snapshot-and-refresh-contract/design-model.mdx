## Meta host invalidation

The LSP **must** treat the following as **hard invalidation** signals for cached compilation context and any meta host schedule keyed to the prior generation:

- Writes to **`Project.proj`**, **`Workspace.proj`**, or workspace **lockfiles** that can change dependency roots or `Meta` attachment.
- Changes to **`project.meta`** on any resolved `Meta` project.
- Successful application of **emit** contributions that alter on-disk sources when the host is configured to persist generated sources (if ever supported); when emit targets only in-memory merged syntax, invalidation follows the document generation counter instead.

**Soft invalidation** (re-run meta process without full workspace graph rebuild) is allowed only when the host can prove the same capture keys remain valid; otherwise the server **must** fall back to hard invalidation. Exact soft rules live under **[Incremental scheduling and determinism](/platform-spec/compiler/metaprogramming-mod-sdk/incremental-scheduling-determinism/)**.

## Practical guidance
When changing behavior in this area, update both the contract text and the corresponding compiler/runtime or tooling tests in the same change so the specification remains executable.
