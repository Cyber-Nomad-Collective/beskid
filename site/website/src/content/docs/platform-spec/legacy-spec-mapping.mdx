<SpecSection title="What this page is for" id="purpose">
This document is **non-normative**. It is a **bridge**: it explains how the former flat `/spec/` chapter layout and large parts of the **`compiler/`** tree line up with the canonical **[platform specification](/platform-spec/)**—especially **[language-meta](/platform-spec/language-meta/)**.

**Normative prose** now lives in platform-spec feature pages (including [Dependency injection](/platform-spec/language-meta/composition/dependency-injection/)). **Bridge pages** describe mapping and migration context: ownership, **status** on [feature](/platform-spec/language-meta/type-system/types/) leaves (how implementation lines up with the standard—not a task list), `relatedTopics`, and navigation.

**Versioning** is **Git**: each merge to `main` advances the language and docs together (rolling release). There is **no** URL segment for a release line under language-meta—only **domain → area → feature** paths.
</SpecSection>

<SpecSection title="Hierarchy vs components vs relations" id="model">
The **navigation tree** (sidebar and graph) is intentionally shallow and predictable:

**Domain → area → feature**

- **Domain** — Top slice of the platform ([language-meta](/platform-spec/language-meta/), [compiler](/platform-spec/compiler/), [tooling](/platform-spec/tooling/), …).
- **Area** — A theme inside the domain ([Type system](/platform-spec/language-meta/type-system/), [Composition](/platform-spec/language-meta/composition/), …).
- **Feature** — A **leaf** tied to a normative chapter (or a narrow slice). Carries **status**, owners, and a link to the canonical spec chapter.

**Component** (in the architecture sense) means a **functional grouping of features**—several features that together serve one capability (e.g. “type shape & algebra”, “inference & dispatch”). Optional **`specLevel: component`** hub pages can appear later; until then, use prose, tables like the one below, or `relatedTopics`.

| Illustrative component (Type system area) | Role | Example features |
| --- | --- | --- |
| Type shape & algebra | Definitions and data shapes users write | [types](/platform-spec/language-meta/type-system/types/), [enums-and-match](/platform-spec/language-meta/type-system/enums-and-match/) |
| Inference & dispatch | How types are inferred and how calls resolve | [type-inference](/platform-spec/language-meta/type-system/type-inference/), [method-dispatch](/platform-spec/language-meta/type-system/method-dispatch/) |

**Relations** (graph edges, `relatedTopics`, links to [Execution](/execution/)) sit **beside** the tree: use them for “see also / depends on / implemented by”, not as a second parent hierarchy.
</SpecSection>

<SpecSection title="How to use this bridge" id="how-to-use">
- **Readers** — Open **feature** pages for status and owners, then the linked **Language Spec** chapter for rules.
- **Authors** — Add or adjust **feature** leaves under the right **area**; use **`relatedTopics`** and the graph for cross-domain ties. When you introduce **component** hubs, name them by **function**.
- **Tooling** — `packages/trudoc/scripts/verify-platform-spec-frontmatter.mjs` lists allowed `specLevel` values and enforces **status** on features.

Older label *“Legacy spec mapping”* referred to a flat path table; this page keeps that inventory with **domain → area → feature** paths and **Git** as the only version line.
</SpecSection>
