# Beskid Projects

This document defines how a Beskid project is structured and what is considered part of the project contract.

## Canonical project manifest

- File name: `Project.proj`
- Location: project root directory
- Format: declarative HCL-based manifest

`Project.proj` is the single source of truth for:

- project identity,
- targets,
- dependencies,
- source roots and build-relevant metadata.

## Canonical project layout

Minimum expected layout:

```text
MyProject/
├── Project.proj
├── src/
│   └── main.bd   # or library entry source
├── obj/
│   └── beskid/   # materialized dependency/build artifacts
└── Project.lock  # generated lockfile
```

Notes:

- `obj/beskid/` is tool-managed output and should not be hand-edited.
- `Project.lock` is generated by resolve/build flows.

## Project model (conceptual)

- **Project**: one root manifest + one workspace root.
- **Target**: a buildable unit (for example executable or library).
- **Dependency**: a typed edge in the project graph.

Dependency sources supported by contract:

- `path` (active)
- `git` and `registry` are reserved extension points in schema/runtime design.

## Determinism and resolution contract

- Dependency resolution must produce a deterministic graph.
- Cycles are invalid.
- Build/check/run operations consume the same resolved graph model.
- Compilation order follows topological dependency order.

## Tooling contract

- CLI, analysis, and LSP should rely on one shared project-loading and resolution model.
- Manifest validation diagnostics should be stable and actionable.
- Formatting and migration tooling may transform manifest text, but must preserve semantic meaning.
