# Platform spec UI components

## Layout contracts (`layout.json`)

- Each hub (`index.mdx`) under `src/content/docs/platform-spec/**/` has a sibling **`layout.json`** validated by Zod (`trudoc` package: `packages/trudoc/src/layout/schema.ts`).
- Presets (`domain-default`, `area-default`, `feature-contract-default`, …) live in `packages/trudoc/src/layout/presets.ts`; node files set `extends` and optional overrides (sections, `minSpecSections`, widgets).
- **Structured document**: optional **`documentStructure`** — `orderedSpecSectionIds` and/or **`orderedSequence`** (`specSection` | `markdownHeading` | `component`) to enforce **order** and presence in CI.
- **Child articles** (feature hubs only): optional **`childArticles`** — `minDirectArticles`, `requireYamlTitle` (YAML `title` non-empty) so PRs can block incomplete bundles.
- **Hub MDX props**: known components (`DomainTiles`, `SpecPageHeader`, `SpecSection`, …) are validated via Zod in `packages/trudoc/src/layout/mdxHubComponents.ts` (extend `HUB_COMPONENT_PROP_SCHEMAS` for new components).
- **PR / CI gate**: `bun run verify:platform-spec-ci` runs frontmatter checks + layout/completeness/MDX/component validation (see repo `.github/workflows/platform-spec-contracts.yml`). Also part of `prebuild`.
- **Report**: `pnpm verify:platform-spec-layout` refreshes `src/generated/platform-spec-layout-report.json` (shown in the **Completeness** reader tab).
- **Area guidance**: see `packages/trudoc/src/layout/AREA_LAYOUT_RECOMMENDATIONS.md`.

## Reader chrome

- **`SpecPageHeader`** — Use on feature hub and similar `specLevel` pages (with explicit owner/submitter props).
- **`SpecArticleChrome`** — Use on `specLevel: article` pages: one line after frontmatter imports the component and renders `<SpecArticleChrome />` (no props; owner/submitter come from frontmatter).
- **`SpecReaderShell`** — Starlight `<Tabs>` for Current document / Articles / History / Architecture (optional) / **Completeness**; still relocates MDX siblings into the first tab via `platform-spec-doc-layout.ts`.
- **`SpecCompletenessReport`** — Renders the current node’s row from the generated layout report.
- **`SpecWidgetRenderer`** — Maps validated `layout.json` widget specs (e.g. `domainTiles`, `twoColumn`) to Astro components for DRY hub rendering.

## GitHub links

- **`GithubResourceLink`** — Link to `blob` / `tree` / `raw` on GitHub. Import in MDX and pass `path` (repo-relative, e.g. `compiler/crates/foo.rs`).
- **`RepoFileCard`** — Card with link plus optional build-time snippet (`snippetLines`, `path` must start with `compiler/` or `site/website/`).

## Markdown `repo` fence

In any `.md` / `.mdx` processed by this site’s Markdown pipeline:

````md
```repo
path: compiler/crates/beskid_cli/src/lib.rs
label: CLI crate
kind: blob
ref: main
```
````

## Git history tab

History data is generated by `bun run generate:platform-spec-git-meta` (also run in `prebuild`). CI should use a full `git` history (`fetch-depth: 0`) when building the site.
